Detailed Description

support for async (asynchronous) programming

The UML below shows the basic structure of the async namespace

The central concept is an t_event. A t_event is a struct that contains data about an interesting event in the application. It can be, for example, a incoming message from a network connection, or a user menu choice.

A handler_t is a function class that will handle a t_event. We can see in the tenacitas::async::handler_t signature that it also receives a std::share_ptr<bool> object. Through this pointer a handler_t function may be informed that its execution took more time than specified, so that it can make the decision to stop running.

A handling_t defines a type of handling for a t_event, and can group as many handler_t objects as necessary. For example, a user menu choice can generate a log message, a message sent to a remote program, and to change the color of widget in the user interface. Each one of this actions is a handling_t. Besides, it is possible to define one handler for the logging handling, five handlers to the network message handling, and two for the widget color changing. When the user makes its menu choice, the five handler_t objects in the message network handling_t will "fight each other" to get the t_event to handle.

Each handling_t defines a timeout applies to all its handler_t objects, and it is possible to define a tenacitas::async::priority for a handling_t, so that the higher priority will receive a t_event before the others handling_t.

The dispatcher_t class is responsible for managing the creation of handling_t objects, adding handler_t objects to a handling_t, and dispatching t_event objects to the handing_t.

In order to make it easier to deal with these classes, the async namespace implements a set of functions: tenacitas::async::add_handling, tenacitas::async::add_handler, tenacitas::async::set_priority and tenacitas::async::dispatch, with some variations.

The sleeping_loop_t allows a function to be executed in a defined interval. It is usefull, among others, to dispatch t_event, in order to simulate an actual event generation and handling.

Please, look at the Examples section for examples on how to use these functions and classes.

template<typename t_time , typename t_function , typename... t_params>
result_traits< std::invoke_result_t<t_function, type::ptr<bool>, t_params...> >::result tenacitas::async::execute ( t_time  p_timeout,
t_function &  p_function,
t_params &&...  p_params 
)

Synchronous function with collaborative timeout control.

Template Parameters
t_timetype of time used to define timeout. It must be one of the defined in std::chrono, such as std::chrono::seconds
t_functionis the type of function that will be executed synchronously. It must have this signature:
return-type operator()(std::shared_ptr<bool>, params-type...)

Where return-type is the type returned by the function. It can void, or any other type or class.

Where std::shared_ptr<bool> is a pointer to bool that will become true if the function being executed exceeds the time limit to execute. It is how the execute function tells the function being executed that a timeout occurred. So, the function being executed should, as much as possible, during its execution, to check for the value of this pointer. If it becomes true, the function must return, as its execution is no longer necessary, neither its return, if any, is expected.

Where params-type is a variadic number of parameters, if any, as the function being executed needs.

Parameters
p_timeoutis the amount of time the function being executed has to finish its execution
p_functionis the function being executed, with the signature described above
p_paramsare the parameters expected by p_function, if any
Returns
If p_function returns, say a int16_t, execute returns std::optional<int16_t>, which will contain a int16_t if p_function executes in less than p_timeout time, i.e., no timeout occurrs; otherwise no value is returned in the std::optional.

However, if p_function's return is void, then execute returns a bool, which is true if no timeout occurrs; or false otherwise.
